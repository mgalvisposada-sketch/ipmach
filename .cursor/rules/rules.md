Rule 1: Use English Only All code, comments, variables, and documentation must be written strictly in English. Do not use any other language, even for temporary or internal variable names. This ensures clarity, collaboration, and consistency across international teams. Rule 2: Provide Complete and Functional Code Always generate full, runnable code examples. Avoid partial implementations, placeholders like TODO, or pseudocode. The code should be ready to run and reflect a production-ready mindset. Rule 3: Apply Best Practices Use widely accepted software engineering principles: SRP (Single Responsibility Principle), DRY (Don't Repeat Yourself), KISS (Keep It Simple, Stupid), YAGNI (You Aren't Gonna Need It). Structure code for clarity, extensibility, and minimal side effects. Rule 4: Validate and Sanitize Input Never assume that input is safe. Always validate user input and sanitize it to prevent errors and security vulnerabilities like XSS, injection attacks, or data corruption. Rule 5: Secure by Default Follow secure coding practices by default. Avoid hardcoded credentials, unsafe eval calls, or dynamic code execution. Consider OWASP Top 10 as a baseline. Rule 6: Use Descriptive Naming Variable, function, and class names must be self-explanatory and written in proper English. Avoid one-letter names or abbreviations unless universally understood (i, id, URL). Rule 7: Handle Errors Gracefully All asynchronous or risky operations must include proper error handling using try/catch, conditional checks, or fallback mechanisms. Avoid silent failures or throwing generic errors. Rule 8: Include Testability Prefer patterns that allow easy testing. Suggest or include unit tests when appropriate. Code shouldbe structured so that it can be tested independently without relying on global state. Rule 9: Explain When Needed When there are multiple valid solutions, briefly explain why you chose one. If the implementation has trade-offs (performance, complexity, security), they must be mentioned. Rule 10: Avoid Bad Practices Do not use magic numbers or hardcoded values, leave debugging code in production, ignore formatting conventions, use outdated methods, introduce unnecessary dependencies, return inconsistent data types, skip input/output validation, repeat code, leave unused imports, or make unchecked assumptions. Rule 11: Keep Functions Small and Focused Each function should do one thing and do it well. If a function grows beyond 30-40 lines or handles multiple concerns, split it. Rule 12: Respect the Project's Structure Follow the existing file and folder structure. Place code in correct layers (services, utils, components, etc.) and respect architectural boundaries. Rule 13: Avoid State Mutations Prefer immutability where possible. Avoid modifying objects or arrays directly unless necessary. Use tools like the spread operator. Rule 14: Use Consistent Code Style Use a consistent code style across all files. Prefer tools like Prettier or ESLint. Avoid mixing tabs and spaces or quote styles. Rule 15: Favor Modern Syntax Use modern JavaScript/TypeScript features like const, let, optional chaining, nullish coalescing, async/await. Avoid var and outdated patterns. Rule 16: Keep Performance in Mind Avoid expensive operations inside loops or renders. Memoize when needed. Debounce or throttle where applicable.Rule 17: Don't Reinvent the Wheel If a reliable, lightweight library exists, use it. Don't reimplement known solutions unless absolutely necessary. Rule 18: Write for Humans First, Machines Second Prioritize readability. Future developers should understand your code instantly. Rule 19: Avoid Over-Abstraction Don't abstract too early. Wait until patterns repeat at least three times. Rule 20: Use Meaningful Commits Commit messages should describe what the code does. Use prefixes like feat:, fix:, refactor: if possible.
